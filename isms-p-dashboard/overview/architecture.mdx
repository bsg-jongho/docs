## 전체 데이터 흐름

<Info>
  **매일 02:00 KST** → 스케줄러가 Step Functions를 시작 → 계정별 병렬 스캔 → 데이터 변환 → 대시보드 반영
</Info>

| 단계 | 구성 요소 | 역할 |
| --- | --- | --- |
| ① 트리거 | EventBridge Scheduler | 매일 02:00 KST에 스캔 시작 |
| ② 오케스트레이션 | Step Functions | 전체 스캔 과정을 순서대로 실행하고 관리 |
| ③ 데이터 수집 | Lambda (3종) | 컴플라이언스 스캔, Security Hub, CloudTrail 병렬 수집 |
| ④ 저장 | S3 (raw/) | 수집된 원본 JSON 저장 (90일 보존) |
| ⑤ 변환 | Glue ETL | JSON → Parquet 변환, 일일 변경 분석 |
| ⑥ 저장 | S3 (processed/) | 변환된 분석용 데이터 저장 (영구 보존) |
| ⑦ 쿼리 | Athena (8개 뷰) | 변환된 데이터를 SQL로 조회 |
| ⑧ 시각화 | QuickSight (9개 시트) | DIRECT_QUERY로 최신 데이터 실시간 표시 |

## 구성 요소별 역할

### EventBridge Scheduler (스케줄러)

| 항목 | 값 |
| --- | --- |
| 역할 | 매일 정해진 시간에 스캔을 시작하는 트리거 |
| 기본 설정 | 매일 02:00 KST |
| 변경 방법 | [스캔 스케줄 설정](/isms-p-dashboard/settings/schedule) 참고 |

<Info>
  **왜 새벽 2시인가:** 전날 발생한 보안 이벤트가 AWS 서비스에 모두 반영된 이후 수집하기 위함입니다. CloudTrail과 Security Hub 모두 이벤트 전파에 수 시간이 걸릴 수 있습니다.
</Info>

### Step Functions (오케스트레이션)

전체 스캔 과정을 순서대로 실행하고 관리합니다.

**실행 순서:**

| 순서 | 단계 | 설명 |
| --- | --- | --- |
| 1 | 계정 목록 로드 | S3에서 `accounts.json` 읽기 |
| 2 | 스캔 역할 확인 | 대상 계정의 ScanRole 존재 여부 확인 |
| 3 | 계정별 병렬 스캔 | 최대 5개 계정 동시 처리, 각 계정에서 3개 수집기 동시 실행 |
| 4 | Glue ETL 변환 | 원본 JSON을 분석용 Parquet으로 변환 |
| 5 | 완료 | 전체 타임아웃: 2시간 (7,200초) |

<Info>
  각 스캔 작업은 실패 시 최대 2회 재시도됩니다 (30초 간격, 2배 백오프). 전체 실행이 실패하면 [알림](/isms-p-dashboard/settings/alerts)이 발송됩니다.
</Info>

### Lambda (데이터 수집기)

3개의 수집기가 각 계정에서 **병렬로** 실행됩니다.

| 수집기 | 수집 대상 | 소요 시간 |
| --- | --- | --- |
| 컴플라이언스 스캔 | ISMS-P 보안 점검 결과 (PASS/FAIL) | 5-10분/계정 |
| Security Hub 수집 | GuardDuty, Inspector 등 탐지 결과 | 2-5분/계정 |
| CloudTrail 수집 | 보안 관련 API 호출 이력 | 2-5분/계정 |

모든 수집기는 대상 계정의 `ScanRole`을 위임(AssumeRole)하여 **읽기 전용**으로 데이터를 수집합니다. 대상 계정의 리소스를 수정하지 않습니다.

### Glue ETL (데이터 변환)

수집된 원본(JSON)을 분석용 형식(Parquet)으로 변환합니다.

**변환 내용:**
- 중첩된 JSON 구조를 평면화하여 쿼리 성능 최적화
- 날짜/계정별 파티셔닝 적용
- **일일 변경 분석:** 전일 대비 신규(NEW), 해결(FIXED), 재발(REGRESSION) 항목 자동 분류

### Athena + QuickSight (쿼리 및 시각화)

- **Athena:** 변환된 데이터를 SQL로 조회하는 서버리스 쿼리 엔진. 8개의 뷰를 제공
- **QuickSight:** 최종 시각화. **DIRECT_QUERY** 방식으로 항상 최신 데이터 표시 (캐시 없음)

## 크로스 계정 접근 방식

<Info>
  대시보드는 **중앙 계정**에 설치되며, **대상 계정**의 데이터를 읽기 전용으로 수집합니다. 각 대상 계정에 `ISMS-P-ScanRole`이 필요합니다.
</Info>

| 항목 | 설명 |
| --- | --- |
| 접근 방식 | 중앙 계정 Lambda → AssumeRole → 대상 계정 ScanRole |
| ExternalId | `isms-p-{중앙계정ID}` 형식 (Confused Deputy 공격 방지) |
| 역할 이름 | `ISMS-P-ScanRole` (변경 가능) |
| 권한 | 읽기 전용. 대상 계정의 리소스를 수정하지 않음 |

## 데이터 저장 구조 (S3)

| 경로 | 용도 | 보존 기간 |
| --- | --- | --- |
| `raw/` | 수집된 원본 JSON | 90일 |
| `processed/` | 변환된 Parquet (대시보드가 참조) | 영구 보존 |
| `config/accounts.json` | 스캔 대상 계정 목록 | - |
| `athena-results/` | Athena 쿼리 결과 캐시 | 7일 |
