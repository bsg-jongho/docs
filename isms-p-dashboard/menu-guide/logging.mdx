<Warning>
  **로깅은 다른 모든 보안 도메인의 기반 인프라입니다.** 로깅이 미설정되면 보안 사고를 탐지할 수 없고, 사후 분석도 불가능합니다. 다른 보안 도메인에 비해 로깅은 높은 준수율이 실질적으로 중요한 영역입니다.
</Warning>

## 다른 시트에 미치는 영향

로깅 시트의 FAIL은 다른 시트의 **데이터 신뢰성에 직접 영향**을 줍니다:

- CloudTrail 비활성 → CloudTrail 이벤트 수집이 불완전
- VPC Flow Logs 미활성 → 네트워크 보안 사각지대
- S3 접근 로깅 미활성 → 데이터 유출 여부 확인 불가

## 핵심 KPI

### CloudTrail 멀티리전 활성화

<Warning>
  **CloudTrail은 AWS 보안 모니터링의 최하위 기반 계층입니다.** CloudTrail이 모든 리전에서 활성화되지 않으면, 이 대시보드를 포함한 모든 보안 도구의 데이터 수집이 근본적으로 불완전해집니다.
</Warning>

AWS CloudTrail은 계정 내에서 발생하는 **모든 AWS API 호출을 기록하는 감사 추적 시스템**입니다. EC2 인스턴스 생성, IAM 권한 변경, S3 버킷 접근, 보안 그룹 수정 등 AWS에서 일어나는 모든 활동의 원본 기록이 CloudTrail에 남습니다.

**멀티리전 활성화가 필수인 이유:**

- CloudTrail은 기본적으로 **설정된 리전의 API 호출만 기록**합니다. 멀티리전이 비활성화된 상태에서 공격자가 비주력 리전(예: ap-northeast-2만 사용하는 환경에서 us-east-1)에 리소스를 생성하면, 해당 활동은 어디에도 기록되지 않습니다.
- GuardDuty, Config, Security Hub 등 상위 보안 서비스도 CloudTrail 이벤트에 의존합니다. CloudTrail이 없는 리전에서는 **위협 탐지 자체가 불가능**하거나, 탐지가 지연될 수 있습니다.
- 이 대시보드의 CloudTrail 수집기(Collector) 역시 CloudTrail이 기록하지 않은 이벤트는 수집할 수 없습니다. 즉, **대시보드에 표시되는 데이터의 완전성이 CloudTrail 멀티리전 설정에 직접 의존**합니다.

<Note>
  **위협 탐지 적시성에 대한 참고:** 운영 매뉴얼에서는 Critical 취약점 대응 기한을 24시간으로 정의하고 있으나, GuardDuty Critical 등급 위협은 실무적으로 1시간 이내 대응이 권고됩니다. CloudTrail 로깅에 공백이 있으면 GuardDuty의 위협 탐지 자체가 지연되거나 누락될 수 있어, 대응 적시성 기준을 충족하기 어려워집니다.
</Note>

**ISMS-P 2.9.1 관점:** 정보보호 및 개인정보보호 관리체계 인증 기준은 "정보시스템에 대한 접근 및 사용에 대해 로그를 기록하고 안전하게 보관"할 것을 요구합니다. 단일 리전 CloudTrail은 전체 AWS 환경을 커버하지 못하므로 이 요건을 충족하지 않습니다.

| 상황 | 심각도 | 조치 |
| --- | --- | --- |
| 모든 계정, 모든 리전에서 CloudTrail 멀티리전 활성화 | - | 정상. 주기적으로 Trail 상태 점검 |
| 일부 계정에서 멀티리전 미활성화 | Critical | 즉시 활성화. 해당 계정의 기존 보안 데이터 신뢰성 재검토 |
| CloudTrail 자체가 비활성화된 계정 존재 | Critical | 최우선 조치. 해당 계정의 모든 보안 판단이 무효 |

<Info>
  다른 KPI와 달리, CloudTrail 멀티리전 활성화는 **예외를 허용하기 어려운 항목**입니다. 개발/테스트 계정이라 하더라도 CloudTrail은 활성화되어야 합니다. 비용이 우려되는 경우 S3 저장소 수명 주기 정책으로 보존 기간을 조정하는 것이 올바른 접근입니다.
</Info>

### 로깅 준수율

로깅은 보안 포렌식의 기반이므로 다른 도메인보다 높은 준수율이 중요합니다. 다만, 모든 환경에서 100%가 필수는 아닙니다. 개발/테스트 환경이나 대량 트래픽의 비민감 서비스는 비용 대비 효과를 고려하여 의도적으로 로깅을 제외할 수 있습니다.

| 상황 | 심각도 | 조치 |
| --- | --- | --- |
| 프로덕션 환경 핵심 서비스 로깅 완비 | - | 제외 항목에 대한 근거 문서화 확인 |
| 프로덕션 일부 서비스 로깅 누락 | High | 누락된 서비스 식별 후 중요도 순 설정 |
| 프로덕션 핵심 서비스(CloudTrail, VPC Flow Logs 등) 로깅 미설정 | Critical | 즉시 조치 필요. 사고 발생 시 원인 분석 불가 |

<Info>
  준수율 수치 자체보다 **"무엇이 로깅되지 않고 있는가"**가 중요합니다. 70%라도 프로덕션 핵심 시스템이 모두 커버되고, 나머지 30%가 개발 환경이라면 합리적일 수 있습니다. 반대로 95%라도 CloudTrail이 빠져 있다면 심각한 문제입니다.
</Info>

<Info>
  **ISMS-P 영향:** 로깅 미비는 다수 인증 기준에 동시에 영향을 줍니다.
  - 2.9.1 로그 및 접속기록 관리
  - 2.9.2 로그 및 접속기록 점검
  - 2.11.1 침해사고 대응 (로그 없이 대응 불가)
  - 2.11.3 사후관리 (로그 없이 원인 분석 불가)
</Info>

### 로깅 실패 건수

로깅 관련 점검에서 FAIL인 총 건수입니다.

## 주요 점검 항목

| 점검 항목 | 설명 | 심각도 | FAIL 시 영향 |
| --- | --- | --- | --- |
| CloudTrail 멀티리전 활성화 | 모든 리전 로깅 여부 | Critical | 특정 리전 API 호출 미기록 |
| 로그 파일 무결성 검증 | CloudTrail 무결성 검증 | High | 공격자의 로그 조작 탐지 불가 |
| S3 접근 로깅 | S3 데이터 접근 기록 | Medium | 데이터 접근 이력 추적 불가 |
| CloudWatch 로그 보존 기간 | 로그 보존 설정 여부 | Medium | 로그가 너무 빨리 삭제되거나 무기한 보존으로 비용 증가 |
| VPC Flow Logs | 네트워크 트래픽 기록 | High | 네트워크 감사 불가 |
