ISMS-P 인증 기준 중 **2.6 접근통제** 영역에 해당합니다.

<Warning>
  네트워크 설정 오류는 가장 빈번한 보안 사고 원인입니다. 보안 그룹 0.0.0.0/0 오픈, 미사용 보안 그룹, VPC Flow Logs 미활성은 즉각적인 침해로 이어질 수 있습니다.
</Warning>

## 핵심 KPI

### 퍼블릭 노출 리소스 수

인터넷에 직접 노출된 리소스의 수입니다.

<Info>
  퍼블릭 노출 리소스 수 자체는 보안 수준을 나타내지 않습니다. ALB, CloudFront, 웹 서버 등 퍼블릭 노출이 **정상적인** 리소스가 대부분의 조직에 존재합니다. ISMS-P에서 중요한 것은 숫자가 아니라 **각 노출이 의도된 것인지, 문서화되어 있는지**입니다.
</Info>

**점검 방법:**

| 확인 항목 | 양호 | 미흡 |
| --- | --- | --- |
| 노출된 리소스가 자산 목록에 등록되어 있는가 | 모든 퍼블릭 리소스가 식별/문서화됨 | 미등록 리소스 존재 |
| 노출 사유가 명확한가 (ALB, API GW 등) | 업무상 필요하여 의도적으로 노출 | 사유 불명 또는 임시 테스트 후 방치 |
| 불필요한 노출이 포함되어 있지 않은가 | 정기적으로 검토하여 불필요 리소스 정리 | 장기간 미검토 |

### 위험 보안 그룹 수

0.0.0.0/0(전체 IP)에서의 인바운드를 허용하는 보안 그룹 수입니다.

모든 0.0.0.0/0이 위험한 것은 아닙니다. **포트에 따라** 판단합니다:

| 포트 | 용도 | 0.0.0.0/0 허용 시 |
| --- | --- | --- |
| 80, 443 | HTTP/HTTPS | **허용 가능** — 웹 서비스는 공개 접근이 정상 |
| 22 | SSH | **Critical** — 무차별 대입 공격의 주요 타겟 |
| 3389 | RDP | **Critical** — 랜섬웨어 공격의 주요 진입점 |
| 3306, 5432 | MySQL, PostgreSQL | **Critical** — 데이터베이스 직접 노출 |
| 모든 포트 | - | **Critical** — 전체 서비스가 인터넷에 노출 |

<Warning>
  SSH(22), RDP(3389), DB(3306/5432) 포트가 0.0.0.0/0으로 열려있으면 **즉시 조치** 필요합니다. 이 포트들은 자동화된 공격의 주요 타겟입니다.
</Warning>

### 미사용 보안 그룹 수

어떤 리소스에도 연결되지 않은 보안 그룹의 수입니다.

| 수치 | 판단 | 조치 |
| --- | --- | --- |
| 0-5 | 양호 | 정기적 정리 |
| 6 이상 | 주의 | 미사용 보안 그룹 정리 권고 |

### VPC Flow Logs 활성화율

| 수치 | 판단 | 근거 |
| --- | --- | --- |
| 100% | 양호 | 모든 VPC의 네트워크 트래픽 감사 가능 |
| 100% 미만 | 경고 | Flow Logs 미활성 VPC는 침해 시 트래픽 분석 불가 |

<Note>
  VPC Flow Logs는 ISMS-P 로그 관리 요구사항의 핵심이므로 원칙적으로 100% 활성화가 필요합니다. 다만 테스트/샌드박스 등 민감 데이터가 없는 VPC를 제외하는 경우, 해당 VPC가 제외 대상임을 **문서화하고 승인**받아야 합니다.
</Note>

<Info>
  **ISMS-P 매핑:** 2.9.1 로그 및 접속기록 관리 — 네트워크 접속 기록을 안전하게 보관하고 주기적으로 점검해야 합니다. Flow Logs가 없으면 이 요구사항을 충족할 수 없습니다.
</Info>

## VPC 보안 위생

### Default VPC 사용 여부

AWS는 모든 리전에 **Default VPC**를 자동으로 생성합니다. Default VPC는 빠른 시작을 위해 설계되었으며, 다음과 같은 **느슨한 보안 기본값**을 가지고 있습니다:

- 모든 서브넷이 퍼블릭 서브넷으로 설정됨 (자동 퍼블릭 IP 할당)
- Default Security Group이 동일 그룹 내 모든 트래픽을 허용
- Default Network ACL이 모든 인바운드/아웃바운드 트래픽을 허용
- Internet Gateway가 기본 연결됨

이러한 기본 구성은 프로덕션 워크로드에 적합하지 않습니다.

<Info>
  **ISMS-P 매핑:** 2.6.1 네트워크 분리 — 네트워크를 업무 목적과 중요도에 따라 분리하고, 각 네트워크 간 접근통제를 수행해야 합니다. Default VPC의 단일 평면 구조는 이 요구사항을 충족하지 못합니다.
</Info>

**대시보드에서 Default VPC가 존재한다고 표시되는 경우:**

Default VPC가 존재하는 것 자체는 AWS가 자동으로 생성한 결과이므로 즉각적인 위험은 아닙니다. 핵심은 **Default VPC 안에 실제 리소스가 배포되어 있는지** 여부입니다.

| 상태 | 위험도 | 조치 |
| --- | --- | --- |
| Default VPC가 존재하나 리소스 없음 | 낮음 | 가능하면 삭제, 불가 시 문서화 |
| Default VPC에 EC2, RDS 등 리소스가 배포됨 | **높음** | 커스텀 VPC로 마이그레이션 필요 |
| Default VPC를 삭제한 상태 | 양호 | 유지 |

<Warning>
  Default VPC에서 운영 중인 리소스가 있다면, 네트워크 분리가 되지 않은 상태입니다. 커스텀 VPC를 설계하여 퍼블릭/프라이빗 서브넷을 분리하고, 워크로드를 마이그레이션하는 것을 권고합니다.
</Warning>

**조치 가이드:**

1. 각 리전의 Default VPC에 배포된 리소스를 확인합니다 (EC2, RDS, Lambda, ELB 등)
2. 리소스가 있는 경우, 적절한 서브넷 설계(퍼블릭/프라이빗/DB 서브넷)를 갖춘 커스텀 VPC를 생성합니다
3. 리소스를 커스텀 VPC로 마이그레이션합니다
4. 마이그레이션 완료 후, 가능하면 Default VPC를 삭제합니다
5. Default VPC를 삭제할 수 없는 경우(조직 정책 등), 해당 사유를 문서화합니다

### Default Security Group 규칙

모든 VPC에는 삭제할 수 없는 **default** 이름의 보안 그룹이 존재합니다. 이 보안 그룹은 리소스를 생성할 때 명시적으로 보안 그룹을 지정하지 않으면 자동으로 할당됩니다.

보안 모범 사례(CIS AWS Foundations Benchmark)에 따르면, Default Security Group에는 **인바운드/아웃바운드 규칙이 없어야** 합니다.

<Note>
  Default Security Group에 규칙이 존재한다는 것은, 누군가가 목적별 보안 그룹을 생성하지 않고 기본 보안 그룹을 그대로 사용하고 있다는 의미입니다. 이는 보안 그룹 관리 체계가 미흡하다는 신호입니다.
</Note>

**대시보드에서 Default Security Group에 규칙이 있다고 표시되는 경우:**

| 확인 항목 | 양호 | 미흡 |
| --- | --- | --- |
| Default SG에 인바운드 규칙이 없는가 | 모든 인바운드 규칙 제거됨 | 인바운드 규칙 존재 |
| Default SG에 아웃바운드 규칙이 없는가 | 모든 아웃바운드 규칙 제거됨 | 아웃바운드 규칙 존재 |
| Default SG에 연결된 리소스가 없는가 | 연결된 리소스 없음 | EC2, RDS 등이 Default SG 사용 중 |

<Info>
  **ISMS-P 매핑:** 2.6.2 접근통제 — 정보시스템과 네트워크에 대한 접근은 업무 목적에 따라 최소한의 범위로 제한해야 합니다. Default Security Group을 그대로 사용하면 접근통제 정책이 명확히 정의되지 않은 것으로 간주될 수 있습니다.
</Info>

**조치 가이드:**

1. Default Security Group의 모든 인바운드/아웃바운드 규칙을 제거합니다
2. Default Security Group에 연결된 리소스가 있다면, 해당 리소스의 역할에 맞는 전용 보안 그룹을 생성합니다
3. 리소스의 보안 그룹을 새로 생성한 전용 보안 그룹으로 변경합니다
4. 향후 리소스 생성 시 반드시 목적에 맞는 보안 그룹을 명시적으로 지정하도록 운영 절차를 수립합니다

<Warning>
  Default Security Group의 규칙을 제거하기 전에, 해당 보안 그룹을 사용 중인 리소스가 없는지 반드시 확인하십시오. 사용 중인 리소스가 있는 상태에서 규칙을 제거하면 서비스 장애가 발생할 수 있습니다.
</Warning>
