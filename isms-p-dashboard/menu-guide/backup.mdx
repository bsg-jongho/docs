<Warning>
  백업은 보안 사고 및 장애 발생 시 비즈니스 연속성을 보장합니다. 랜섬웨어 공격 시 백업이 없으면 데이터 복구가 불가능하며, 실수로 인한 삭제도 복구할 수 없습니다.
</Warning>

## 핵심 KPI

### 백업 준수율

백업 준수율은 단순히 높은 숫자를 추구하는 지표가 아닙니다. **핵심 자산이 보호되고 있는지**, 그리고 미보호 리소스에 대한 **의사결정이 문서화되어 있는지**가 중요합니다.

| 상황 | 판단 | 조치 |
| --- | --- | --- |
| 핵심 자산(프로덕션 DB, 재생성 불가 데이터) 100% 보호 | 양호 | 제외된 리소스의 근거 문서화 확인 |
| 핵심 자산 일부 미보호 | 경고 | 미보호 핵심 자산 즉시 식별 및 백업 설정 |
| 핵심 자산 다수 미보호 또는 제외 근거 미문서화 | 위험 | 자산 분류 기준 수립 후, 핵심 자산부터 백업 정책 적용 |

<Warning>
  백업 준수율이 80%라도 핵심 자산이 모두 보호되고 있다면 적절할 수 있습니다. 반대로 95%라도 누락된 5%가 프로덕션 DB라면 심각한 문제입니다. **숫자보다 "무엇이 빠져 있는가"를 확인하세요.**
</Warning>

### 복원력 비율

멀티-AZ 배포, 자동 장애 조치 등 가용성 관련 점검의 준수율입니다.

| 구성 | 가용성 | 비용 대비 |
| --- | --- | --- |
| 단일-AZ | 99.5% | 1x |
| 멀티-AZ | 99.95% | 1.5-2x |
| 멀티-리전 | 99.99% | 3-5x |

<Info>
  ISMS-P는 "적절한" 재해 복구 계획을 요구하지, 반드시 멀티-AZ를 강제하지는 않습니다. **RTO/RPO 목표 대비 현재 구성이 적절한지**를 평가하세요.
</Info>

### 백업 로깅 준수율

백업 관련 로깅 및 감사 추적(Audit Trail)이 적절히 구성되어 있는지를 나타내는 비율입니다. 백업 자체도 중요하지만, ISMS-P 관점에서는 **백업이 실제로 수행되었는지 증명**하고, **누가 백업 데이터에 접근했는지 추적**할 수 있어야 합니다.

| 점검 대상 | 설명 | 예시 |
| --- | --- | --- |
| S3 액세스 로깅 | 백업 버킷에 대한 접근 기록 | 백업 S3 버킷의 서버 액세스 로깅 활성화 여부 |
| CloudTrail 로깅 | 백업 작업에 대한 API 호출 기록 | AWS Backup, S3 PutObject 등 백업 관련 API 추적 |
| RDS 감사 로깅 | 데이터베이스 백업 작업 기록 | RDS 감사 로그(audit log) 활성화 여부 |

| 상황 | 판단 | 조치 |
| --- | --- | --- |
| 모든 백업 리소스에 로깅 활성화 (95% 이상) | 양호 | 로그 보존 기간이 규정 요건 충족하는지 확인 |
| 주요 백업 리소스에 로깅 미활성화 (85~95%) | 경고 | 미활성화 리소스 식별 후 로깅 설정 적용 |
| 다수 백업 리소스에 로깅 미구성 (85% 미만) | 위험 | 백업 로깅 정책 수립 후 전체 적용 |

<Info>
  ISMS-P 인증 심사에서는 "백업을 하고 있다"는 것만으로는 부족합니다. **백업 수행 이력**, **백업 데이터 접근 이력**, **복원 테스트 이력**이 감사 증적으로 제출 가능해야 합니다. 로깅이 없으면 백업을 했더라도 이를 증명할 수 없습니다.
</Info>

<Warning>
  백업 버킷에 대한 액세스 로깅이 비활성화되어 있으면, 내부자가 백업 데이터를 무단 열람하거나 삭제하더라도 추적이 불가능합니다. 특히 개인정보가 포함된 백업 데이터의 경우, 접근 로그 없이는 유출 사고 시 영향 범위 파악 자체가 어려워집니다.
</Warning>

### 미보호 리소스 수

백업이 설정되지 않은 리소스의 수입니다.

| 리소스 유형 | 백업 필요 | 근거 |
| --- | --- | --- |
| 프로덕션 DB (RDS) | **필수** | 비즈니스 핵심 데이터 |
| 사용자 업로드 S3 | **필수** | 재생성 불가 데이터 |
| 설정/코드 S3 | 권장 | Git 등에서 복원 가능하면 선택적 |
| 임시/캐시 데이터 | 불필요 | 재생성 가능 |

<Warning>
  모든 리소스에 백업이 필요한 것은 아닙니다. 이 KPI가 0이 아니라면, 해당 리소스가 **의도적으로 제외**된 것인지 확인하세요. 의도적 제외라면 근거를 문서화하세요.
</Warning>

## 주요 점검 항목

| 점검 항목 | 설명 | 심각도 |
| --- | --- | --- |
| RDS 자동 백업 활성화 | RDS 자동 백업 여부 | High |
| S3 버전 관리 활성화 | S3 버전관리 여부 | Medium |
| DynamoDB PITR 활성화 | 특정 시점 복구 가능 여부 | Medium |
| EBS 스냅샷 생명주기 | 스냅샷 자동화 여부 | Medium |
| RDS 멀티-AZ 배포 | 고가용성 배포 여부 | High |
