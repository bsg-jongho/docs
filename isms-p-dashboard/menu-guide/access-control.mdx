## 이 시트가 중요한 이유

<Warning>
  IAM은 AWS 보안의 기반입니다. 접근 권한이 제대로 관리되지 않으면 **Root 계정 탈취**(전체 통제권 상실), **과도한 권한 부여**(내부자 위협), **오래된 액세스 키**(무단 접근)로 이어질 수 있습니다.
</Warning>

## 핵심 KPI

### Root MFA 비율

Root 계정에 MFA(다중 인증)가 활성화된 비율입니다.

| 수치 | 심각도 | 판단 |
| --- | --- | --- |
| **100%** | - | 정상 |
| **100% 미만** | **Critical** | Root 계정은 AWS 최고 권한이므로 즉시 확인 후 조치. 예외 사유가 인정되기 매우 어려운 항목 |

<Warning>
  **영향도:** Root 계정은 과금 정보 변경, 계정 해지, 모든 리소스 삭제가 가능합니다. MFA가 없는 Root 계정은 비밀번호 유출 시 **복구 불가능한 피해**가 발생할 수 있습니다.

  **ISMS-P 매핑:** 2.5.1 사용자 인증 — "정보시스템에 대한 접근은 안전한 인증절차를 거친 후 허용하여야 한다"
</Warning>

**조치 방법:**
1. AWS 콘솔 로그인 (Root 계정)
2. 우측 상단 계정명 → 보안 자격 증명
3. MFA 디바이스 할당 → 가상 MFA 디바이스 (Google Authenticator 등)
4. QR 코드 스캔 → 연속 2개 코드 입력 → 활성화

### User MFA 비율

콘솔 로그인이 가능한 IAM 사용자 중 MFA가 활성화된 비율입니다.

<Info>
  **모든 IAM 사용자에게 MFA가 필요한 것은 아닙니다.** 서비스 계정, CI/CD 파이프라인 전용 사용자, API 전용 사용자 등 콘솔 접근이 없는 계정은 MFA 대상에서 제외될 수 있습니다. 이 수치를 볼 때는 MFA 미설정 사용자가 **실제로 콘솔 로그인을 사용하는 사람인지** 확인하는 것이 중요합니다.
</Info>

| 수치 | 심각도 | 판단 |
| --- | --- | --- |
| 100% | - | 양호. 단, 콘솔 미사용 계정까지 포함된 경우 과도한 MFA 부여일 수 있음 |
| 80-99% | High | MFA 미설정 사용자 목록을 확인하여, 콘솔 로그인 사용자인지 서비스 계정인지 구분 후 조치 |
| 80% 미만 | Critical | 콘솔 사용자 대다수에 MFA가 없을 가능성. 대상 사용자 분류 후 MFA 필수 정책 도입 검토 |

### 오래된 액세스 키 비율

90일 이상 교체되지 않은 액세스 키의 비율입니다.

| 수치 | 심각도 | 판단 |
| --- | --- | --- |
| 0% | - | 모든 키가 90일 이내 교체됨 |
| 1-20% | Medium | 해당 키 목록을 확인하여 교체 대상인지 예외 대상인지 분류 |
| 20% 초과 | High | 키 관리 프로세스 전반 점검 필요. 자동 교체 메커니즘 도입 검토 |

<Info>
  **왜 90일인가:** 키가 오래될수록 유출 가능 경로가 누적됩니다 (Git 이력, 이전 직원 PC 등). 90일은 업계 표준 기간이며, ISMS-P에서도 주기적 교체를 요구합니다.

  **단, 모든 오래된 키가 문제는 아닙니다.** 레거시 시스템 연동, 외부 파트너 API 연결 등 교체가 즉시 어려운 경우가 있을 수 있습니다. 이런 경우 **예외 사유를 문서화**하고, IP 제한, 최소 권한 정책 등 보완 통제를 적용하고 있는지 확인하세요. 문서화된 예외는 ISMS-P 심사에서 인정받을 수 있습니다.
</Info>

**조치 방법:**
1. 새 액세스 키 생성
2. 애플리케이션에서 새 키로 교체
3. 이전 키 비활성화 → 정상 동작 확인 → 삭제
4. (권장) IAM 역할 기반 접근으로 전환하여 장기 키 사용 최소화

### 미사용 IAM 사용자

90일 이상 콘솔 로그인이나 액세스 키 사용 이력이 없는 IAM 사용자의 수입니다. Prowler 점검 항목 `iam_user_unused_credentials_90_days`에 해당합니다.

<Warning>
  **왜 위험한가:** 미사용 계정은 공격자에게 가장 매력적인 표적입니다. 퇴사자 계정, 프로젝트 종료 후 방치된 서비스 계정, 테스트용으로 만들고 잊은 계정 등은 비밀번호나 키가 유출되어도 아무도 인지하지 못합니다. 실제 보안 사고에서 미사용 계정을 통한 초기 침투가 빈번하게 발생합니다.

  **ISMS-P 매핑:** 2.5.4 접근권한 검토 — "정보시스템에 대한 접근권한을 주기적으로 검토하여 과도하거나 불필요한 권한을 제거하여야 한다"
</Warning>

<Info>
  **이 수치가 0이 아니라고 해서 즉시 문제인 것은 아닙니다.** 중요한 것은 각 미사용 사용자에 대해 **검토가 이루어졌는지** 여부입니다. 미사용 사용자가 발견되면 다음을 확인하세요:

  - 해당 사용자가 여전히 필요한 계정인가? (퇴사자, 부서 이동, 프로젝트 종료 등)
  - 필요 없다면 → 비활성화 후 일정 기간 경과 뒤 삭제
  - 필요하지만 현재 미사용 상태라면 → 사유를 문서화 (예: 분기별 배치 작업 전용 계정)

  문서화된 검토 기록이 있으면 ISMS-P 심사에서 적절한 관리로 인정받을 수 있습니다.
</Info>

**조치 방법:**
1. 미사용 사용자 목록을 HR 또는 팀 리드와 함께 검토하여 현재 재직/활동 중인 인원인지 확인
2. 불필요한 계정은 먼저 **비활성화**(콘솔 비밀번호 삭제, 액세스 키 비활성화)하여 영향도 파악
3. 비활성화 후 2~4주간 문제가 없으면 계정 삭제
4. 예외 계정(분기별 사용 등)은 예외 사유와 보완 통제(IP 제한, 최소 권한 등)를 문서화
5. 이 검토를 분기별 또는 반기별 정기 프로세스로 수립

### IAM 패스워드 정책

IAM 사용자의 비밀번호 정책이 보안 기준에 부합하는지 점검합니다. 대시보드는 다음 항목을 각각 확인합니다:

| 점검 항목 | 기준 |
| --- | --- |
| 최소 길이 | 14자 이상 |
| 복잡도 | 대문자, 소문자, 숫자, 특수문자 각 1개 이상 포함 |
| 만료 기간 | 90일 이내 |
| 재사용 방지 | 최근 24회 이내 비밀번호 재사용 금지 |

<Warning>
  **MFA가 있는데 왜 비밀번호 정책이 필요한가:** MFA는 인증의 한 계층이지 유일한 방어선이 아닙니다. MFA 디바이스 분실, 복구 절차 중 MFA 일시 해제, MFA 미적용 경로(CLI 등)가 존재할 수 있으며, **방어 심층(Defense in Depth)** 원칙에 따라 각 인증 요소가 독립적으로 충분한 강도를 가져야 합니다.

  **ISMS-P 매핑:** 2.5.1 사용자 인증 — "안전한 인증절차"에는 비밀번호 복잡도, 주기적 변경, 재사용 제한이 포함됩니다.
</Warning>

<Info>
  **SSO/Federation을 사용하는 조직의 경우:** IdP(Okta, Azure AD 등)를 통해 로그인하는 사용자에게는 IdP의 비밀번호 정책이 적용됩니다. 대시보드가 점검하는 것은 **AWS IAM 수준의 비밀번호 정책**이므로, IAM 사용자를 직접 사용하지 않는 환경에서는 이 항목이 실질적으로 영향이 적을 수 있습니다. 다만, IAM 사용자가 한 명이라도 존재한다면 해당 정책은 적용되어 있어야 합니다.
</Info>

**조치 방법:**
1. AWS 콘솔 → IAM → 계정 설정 → 비밀번호 정책
2. 위 기준에 맞게 각 항목을 설정 (최소 길이 14, 복잡도 4종, 만료 90일, 재사용 24회)
3. 정책 변경 시 기존 사용자에게 다음 로그인 시 비밀번호 변경을 요구할지 결정
4. SSO 환경이라면 IAM 사용자 사용 현황을 먼저 파악하고, 불필요한 IAM 사용자를 정리하는 것이 우선

### IAM 실패 건수

접근권한 관련 점검에서 FAIL인 총 건수입니다. 수치 자체보다 **어떤 항목이 FAIL인지**가 중요합니다. 하단 상세 테이블에서 개별 FAIL 항목의 리소스와 심각도를 확인하세요.

## 주요 점검 항목

| 점검 항목 | 설명 | 심각도 | FAIL 시 의미 |
| --- | --- | --- | --- |
| Root MFA 활성화 | Root 계정 MFA 여부 | Critical | 단일 인증으로만 보호됨 |
| IAM User MFA | 콘솔 사용자 MFA 여부 | High | 비밀번호만으로 로그인 가능 |
| 액세스 키 교체 | 90일 이내 교체 여부 | Medium | 장기간 무단 접근 가능 |
| Root 액세스 키 미존재 | Root 키 존재 여부 | Critical | 유출 시 최고 권한 탈취 |
| 관리자 권한 미부여 | `*:*` 권한 미부여 | High | 최소 권한 원칙 위반 |
| 인라인 정책 미사용 | 인라인 정책 존재 여부 | Medium | 정책 관리 일관성 저하 |
| 미사용 IAM 사용자 | 90일 이상 미사용 자격 증명 | Medium | 방치된 계정을 통한 무단 접근 가능 |
| IAM 패스워드 정책 | 길이/복잡도/만료/재사용 기준 충족 여부 | Medium | 취약한 비밀번호 허용 상태 |
