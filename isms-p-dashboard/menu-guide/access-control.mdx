## 이 시트가 중요한 이유

<Warning>
  IAM은 AWS 보안의 기반입니다. 접근 권한이 제대로 관리되지 않으면 **Root 계정 탈취**(전체 통제권 상실), **과도한 권한 부여**(내부자 위협), **오래된 액세스 키**(무단 접근)로 이어질 수 있습니다.
</Warning>

## 핵심 KPI

### Root MFA 비율

Root 계정에 MFA(다중 인증)가 활성화된 비율입니다.

| 수치 | 심각도 | 판단 |
| --- | --- | --- |
| **100%** | - | 정상 |
| **100% 미만** | **Critical** | 즉시 조치. Root 계정은 AWS 최고 권한 |

<Warning>
  **영향도:** Root 계정은 과금 정보 변경, 계정 해지, 모든 리소스 삭제가 가능합니다. MFA가 없는 Root 계정은 비밀번호 유출 시 **복구 불가능한 피해**가 발생할 수 있습니다.

  **ISMS-P 매핑:** 2.5.1 사용자 인증 — "정보시스템에 대한 접근은 안전한 인증절차를 거친 후 허용하여야 한다"
</Warning>

**조치 방법:**
1. AWS 콘솔 로그인 (Root 계정)
2. 우측 상단 계정명 → 보안 자격 증명
3. MFA 디바이스 할당 → 가상 MFA 디바이스 (Google Authenticator 등)
4. QR 코드 스캔 → 연속 2개 코드 입력 → 활성화

### User MFA 비율

콘솔 로그인이 가능한 IAM 사용자 중 MFA가 활성화된 비율입니다.

| 수치 | 심각도 | 판단 |
| --- | --- | --- |
| 100% | - | 양호 |
| 80-99% | High | 미설정 사용자 확인 후 MFA 설정 요청 |
| 80% 미만 | Critical | MFA 필수 정책 도입 검토 |

### 오래된 액세스 키 비율

90일 이상 교체되지 않은 액세스 키의 비율입니다.

| 수치 | 심각도 | 판단 |
| --- | --- | --- |
| 0% | - | 양호 |
| 1-20% | Medium | 해당 키 교체 필요 |
| 20% 초과 | High | 자동 교체 메커니즘 도입 필요 |

<Info>
  **왜 90일인가:** 키가 오래될수록 유출 가능 경로가 누적됩니다 (Git 이력, 이전 직원 PC 등). 90일은 업계 표준 기간이며, ISMS-P에서도 주기적 교체를 요구합니다.
</Info>

**조치 방법:**
1. 새 액세스 키 생성
2. 애플리케이션에서 새 키로 교체
3. 이전 키 비활성화 → 정상 동작 확인 → 삭제
4. (권장) IAM 역할 기반 접근으로 전환하여 장기 키 사용 최소화

### IAM 실패 건수

접근권한 관련 점검에서 FAIL인 총 건수입니다. 수치 자체보다 **어떤 항목이 FAIL인지**가 중요합니다. 하단 상세 테이블에서 개별 FAIL 항목의 리소스와 심각도를 확인하세요.

## 주요 점검 항목

| 점검 항목 | 설명 | 심각도 | FAIL 시 의미 |
| --- | --- | --- | --- |
| Root MFA 활성화 | Root 계정 MFA 여부 | Critical | 단일 인증으로만 보호됨 |
| IAM User MFA | 콘솔 사용자 MFA 여부 | High | 비밀번호만으로 로그인 가능 |
| 액세스 키 교체 | 90일 이내 교체 여부 | Medium | 장기간 무단 접근 가능 |
| Root 액세스 키 미존재 | Root 키 존재 여부 | Critical | 유출 시 최고 권한 탈취 |
| 관리자 권한 미부여 | `*:*` 권한 미부여 | High | 최소 권한 원칙 위반 |
| 인라인 정책 미사용 | 인라인 정책 존재 여부 | Medium | 정책 관리 일관성 저하 |
