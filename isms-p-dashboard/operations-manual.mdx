## KPI 읽는 법

대시보드의 각 시트 상단에는 해당 보안 영역의 핵심 지표가 표시됩니다. 이 지표를 어떻게 해석하고, 어떤 순서로 대응해야 하는지를 안내합니다.

### 아침에 대시보드를 열었을 때

매일 출근 후 대시보드를 열면 가장 먼저 **Overview 시트**를 확인합니다. 이때 봐야 할 순서는 다음과 같습니다.

| 순서 | 확인 항목 | 판단 기준 | 다음 행동 |
| --- | --- | --- | --- |
| 1 | Critical/High 실패 건수 | 0이 아니면 즉시 확인 | 해당 도메인 시트로 이동하여 어떤 리소스인지 파악 |
| 2 | 전일 대비 REGRESSION | 1건이라도 있으면 확인 | 이전에 조치한 항목이 왜 다시 실패했는지 원인 파악 |
| 3 | 신규(NEW) 실패 항목 | Critical/High 심각도 우선 | CloudTrail에서 누가 어떤 변경을 했는지 추적 |
| 4 | 전체 준수율 변화 | 전일 대비 하락 여부 | 하락 시 어떤 도메인에서 떨어졌는지 확인 |

<Warning>
  **준수율 숫자에 안심하지 마세요.** 전체 준수율이 97%라도 나머지 3%에 Root 계정 MFA 미설정, 데이터베이스 포트 전체 공개 같은 Critical 항목이 포함되어 있다면 ISMS-P 인증에 심각한 영향을 줄 수 있습니다. 준수율보다 **심각도가 높은 실패 항목이 있는지**를 먼저 확인하세요.
</Warning>

### 지표 유형별 해석

대시보드에 표시되는 지표는 크게 3가지 유형입니다.

**비율(%) 지표** — 특정 보안 영역에서 자동 점검을 통과한 비율

비율이 100%가 아니라고 해서 반드시 문제가 있는 것은 아닙니다. 예를 들어 웹 서비스를 운영하는 보안 그룹이 80/443 포트를 외부에 열어놓은 것은 정상적인 운영 설정입니다. 중요한 것은 **미통과 항목이 의도된 설정인지, 아니면 실제 보안 허점인지를 구분**하는 것입니다.

| 확인 순서 | 질문 | 행동 |
| --- | --- | --- |
| 1 | 전일 대비 비율이 변했는가? | 변했다면 어떤 항목이 바뀌었는지 확인 |
| 2 | 미통과 항목 중 Critical/High가 있는가? | 있다면 해당 리소스와 사유를 즉시 확인 |
| 3 | 미통과 항목이 의도된 설정인가? | 의도된 것이라면 예외 사유를 문서화. 의도하지 않은 것이라면 조치 |

<Info>
  ISMS-P 인증 심사에서 중요한 것은 100% 준수율이 아니라, **미준수 항목에 대해 조직이 인지하고 있고 합리적인 사유가 있는지**입니다. 예외 항목은 "왜 이렇게 설정했는지", "어떤 보완 조치를 했는지"를 문서화해 두세요.
</Info>

**건수 지표** — 주의가 필요한 항목의 수 (예: Critical 실패 건수, 위험 보안 그룹 수)

건수가 0이 아니라고 무조건 문제는 아닙니다. ALB 앞에 둔 보안 그룹이 0.0.0.0/0을 허용하는 것은 자연스러운 설정입니다. 핵심은 **해당 건수가 모두 설명 가능한 것인지** 확인하는 것입니다.

- **전일 대비 증가했다면** — 새로 추가된 항목이 무엇인지 확인. 의도한 변경인지, 실수인지 판단
- **전일 대비 감소했다면** — 조치가 진행되고 있거나 리소스가 삭제된 것. 의도한 변경인지 확인
- **변동 없이 유지된다면** — 이미 인지하고 예외 처리한 항목인지 확인. 방치되고 있는 것은 아닌지 점검

**변경 추이 지표** — 보안 상태가 개선되고 있는지, 악화되고 있는지를 보여줍니다

| 분류 | 의미 | 관리 포인트 |
| --- | --- | --- |
| **NEW** | 오늘 새로 발견된 실패 항목 | 누가, 언제 설정을 변경했는지 확인 |
| **FIXED** | 어제 실패였다가 오늘 통과로 전환 | 조치 완료 확인. 의도한 변경인지 검증 |
| **REGRESSION** | 이전에 통과했다가 다시 실패로 전환 | **가장 중요.** 조치가 원복된 것이므로 근본 원인 파악 필수 |

FIXED가 NEW보다 많으면 보안 상태가 **개선**되고 있는 것이고, NEW가 FIXED보다 많으면 **악화**되고 있는 것입니다. REGRESSION이 반복되면 일시적 조치가 아니라 인프라 코드나 배포 파이프라인 수준에서 근본적으로 해결해야 합니다.

### 심각도별 대응 기준

실패 항목이 발견되면, 심각도에 따라 대응 우선순위와 보고 체계가 달라집니다.

| 심각도 | 의미 | 대응 시한 | 보고 |
| --- | --- | --- | --- |
| **Critical** | 즉시 악용 가능한 보안 취약점. 인증 심사 시 불합격 사유 | 24시간 내 조치 | 보안 담당 책임자에게 즉시 보고 |
| **High** | 단독으로는 즉시 위험하지 않으나 인증 심사 시 지적 대상 | 1주 내 조치 계획 수립 | 주간 보안 회의에서 공유 |
| **Medium** | 보안 강화 권고 사항. 인증 심사 시 개선 권고 가능 | 1개월 내 | 월간 보안 보고서에 포함 |
| **Low** | 모범 사례 수준의 개선 권고 | 분기 내 | 정기 점검 시 검토 |

<Info>
  ISMS-P 인증 심사에서는 Critical과 High 항목이 **결함**으로 지적될 수 있습니다. 각 항목에 대해 조치를 완료했거나, 미조치 시 합리적 사유와 보완 조치가 문서화되어 있어야 합니다.
</Info>

### ISMS-P 인증 기준과 대시보드 매핑

각 대시보드 시트가 ISMS-P 인증 기준의 어떤 영역을 커버하는지 보여줍니다. 인증 심사 준비 시 해당 기준의 기술적 증빙 자료로 활용할 수 있습니다.

| 대시보드 시트 | ISMS-P 인증 기준 | 심사 시 활용 |
| --- | --- | --- |
| 접근권한 관리 | 2.5 인증 및 권한관리 | MFA 적용 현황, 권한 관리 증빙 |
| 네트워크 보안 | 2.6 접근통제 | 네트워크 분리, 접근통제 증빙 |
| 암호화 | 2.7 암호화 적용 | 저장/전송 데이터 암호화 증빙 |
| 로깅/모니터링 | 2.9 로그 및 접속기록 관리 | 감사 로그 운영 증빙 |
| 자산 관리 | 2.1 정보자산 관리 | 자산 현황 및 보안 점검 증빙 |
| 취약점 관리 | 2.10 취약점 점검 및 조치 | 정기 점검 및 조치 이력 증빙 |
| 백업/복구 | 2.12 재해 복구 | 백업 현황 및 복원 능력 증빙 |
| 개선 현황 | 전 영역 | 지속적 보안 관리 활동 증빙 |
